## 0 · Goal & Non-Goals

### 0.1 Goal

Re-architect the existing v1.8.9 markdown-to-PDF Python tool into an **enterprise-grade open-source product** distributed via GitHub, usable from Claude Code, Cursor, Gemini CLI, open-source LLM stacks, CI systems, and Python backends. The product must satisfy four enterprise pillars:

1. **Reliability** — deterministic output mode, version-locked dependencies, 4-layer golden testing, structured errors, atomic file writes, resource limits.
2. **Compatibility** — single core engine consumable by 4 adapters (CLI, MCP server, Skill, GitHub Action) plus a Python API; multi-platform OS matrix; SemVer compatibility guarantees.
3. **Security** — 5-layer watermark model (visible / XMP / steganographic / encryption / signature), brand-lock policy, schema validation, path sandboxing, audit log, Mermaid input lint + process isolation, no-default-network-access posture.
4. **Future extensibility** — pluggable renderer / watermark / encryption / signature interfaces, brand pack and template pack schemas with SemVer compatibility promise, AST middle layer decoupling parsing from rendering.

The product ships with **two orthogonal customisation systems**:

- **Brand kits** answer *who* issued the document (logo, colours, fonts, footer, compliance text).
- **Template packs** answer *what kind* of document this is (sales quote / user manual / training material / analysis report / certificate / …) — defining cover pages, required sections, structured front-matter fields, layout rules, and rendering engine choice.

A rendered PDF is the cross product `brand × template`: any brand can be combined with any template.

### 0.2 Non-Goals (Explicit Scope Cuts)

- **Not** a Word/HTML/EPUB exporter. Only PDF output.
- **Not** a markdown editor / live preview tool. Pure batch conversion.
- **Not** a DRM platform. Watermarks deter and enable forensics; they don't prevent OCR + re-typing.
- **Not** an OS-level screenshot blocker. That is EDR/DLP territory.
- **Not** a PKI implementation. L5 digital signature ships as an interface only; concrete signing requires an enterprise-supplied plugin.
- **Not** vendoring fonts in core wheel. CJK and other scripts are opt-in extras.
- **Not** a quotation-approval / pre-release-audit / wording-assist platform. Templates produce PDFs; downstream commercial-process gating is a separate concern handled by the user's own systems. Templates may declare structural validation rules (required fields, type constraints) but never govern business decisions.

### 1.5 Tech Stack (Pinned)

- **Language:** Python 3.10+ (matrix tested on 3.10–3.13)
- **Markdown:** `markdown-it-py` (CommonMark + GFM plugins). **Replaces** v1.8.9's regex parsing.
- **PDF generation engines (per-template choice):**
  - `reportlab` 4.x — default engine; code-defined layout; best for narrative documents (manual, training, report); proven CJK + watermark support inherited from v1.8.9.
  - `weasyprint` — opt-in HTML+CSS engine (extra `[weasyprint]`, ~30MB pure Python, no Chromium); best for highly designed templates (quote, certificate, invoice) where designers contribute HTML/CSS.
  - `chromium` (deferred) — Playwright-based fallback (extra `[chromium]`); reserved for templates that depend on Chromium-only CSS features. **Not** in v2.0; documented as v2.x extension point.
- **PDF post-processing:** `pikepdf` (preferred for XMP / encryption / linearisation), `pypdf` retained for overlay merging where pikepdf is heavier.
- **Code highlighting:** `pygments` (kept from v1.8.9).
- **Mermaid (3 renderers):**
  - `httpx` for Kroki HTTP calls (default when `KROKI_URL` set)
  - `puppeteer` via `mermaid-cli` (opt-in extra `[puppeteer]`, version-locked Chromium for determinism)
  - `mermaid-py` (pure-Python fallback, opt-in extra `[mermaid-pure]`)
- **Image:** `Pillow` (raster), `cairosvg` (SVG → PNG)
- **Template engine:** `jinja2` with sandboxed environment (no `__import__`, no filesystem ops, no exec) for HTML templates and section partials.
- **Schema validation:** `pydantic` v2 (brand, template, front-matter)
- **Logging:** `structlog`
- **Metrics:** `prometheus_client`
- **CLI:** `click` (replaces argparse; cleaner subcommand support)
- **MCP:** `mcp` (official Anthropic SDK)
- **Testing:** `pytest`, `pytest-asyncio`, `verapdf-cli` (PDF/A validation in CI)
- **Lint:** `ruff` + `mypy --strict`

## 2 · Rendering Pipeline (Data Flow)

### 2.1 Phases

```
Input → Validate → Parse → Asset Resolution → Render → Post-Process → Output
```

#### 2.1.1 Input Phase

`RenderRequest` is the unified input contract for all adapters:

```python
@dataclass(frozen=True)
class RenderRequest:
    source: str | Path                  # markdown content or file path
    source_type: Literal["content", "path"]
    output: Path
    brand: str | InlineBrand | None     # name or dict
    brand_overrides: dict[str, Any]     # field-level overrides (whitelist enforced)
    template: str | InlineTemplate | None  # template id; defaults to "generic"
    template_overrides: dict[str, Any]  # field-level overrides (whitelist enforced)
    front_matter_imports: list[Path]    # extra YAML files merged into front-matter
                                        # (e.g., --prefill, --pricing-config, --issuer)
    strict: bool                        # escalate front-matter warnings → errors
    bundle: list[str]                   # opt-in extras: ["json", "review_md"]
                                        # default [] = pdf only
    watermark: WatermarkOptions         # user, level, custom_text
    pdfa: bool                          # PDF/A-2b mode
    deterministic: bool                 # SOURCE_DATE_EPOCH-respecting
    locale: str                         # zh-CN / en / ja / …
    audit_enabled: bool                 # default True; policy may override
```

Markdown front-matter may also declare `template:`, `brand:`, and arbitrary template-specific fields. CLI arguments and front-matter merge with **CLI > front-matter > template defaults > brand defaults** precedence.

#### 2.1.2 Validate Phase (fail fast on structure; warn on content)

Order matters; each step can short-circuit with structured error:

1. `policy.py`: read `MD_PDF_BRAND_LOCK` and `/etc/md-to-pdf/policy.yaml`; reconcile against request (strictest wins).
2. `brand/registry`: resolve brand id through 3-layer overlay or accept inline brand.
3. `brand/schema`: pydantic v2 validation of resolved brand pack.
4. `brand/safe_paths`: validate brand asset paths.
5. `template/registry`: resolve template id (default `"generic"`) through 3-layer overlay or accept inline template.
6. `template/schema`: pydantic v2 validation of resolved template pack.
7. `template/composer`: produce the effective `RenderConfig` by composing brand × template (template's `layout:` overrides brand's; brand's `compliance` and `assets` win for identity fields). The composer also picks the render engine (`reportlab` or `weasyprint`) declared by the template.
8. `template/imports`: resolve `imports:` declared in template (e.g., `pricing_config`, `issuer`, `prefill`) — paths sandboxed to allowed dirs (policy + working dir).
9. `template/front_matter`: parse markdown front-matter, merge with imports, validate against the template's `front_matter_schema`. Required fields missing → **warning** by default (configurable to error via `--strict` or `policy.template.strict_front_matter: true`); type errors and pattern mismatches always → error.
10. `fonts/manager`: confirm CJK fonts available iff input contains CJK chars; else fail loudly with `FONT_NOT_INSTALLED`.
11. `security/audit`: emit `render.start` log line with input hash, brand, template, policy snapshot, render id (UUIDv4).

#### 2.1.3 Parse Phase

`markdown-it-py` produces tokens → converted to internal AST in `core/markdown/ast.py`. AST node types:

- `Document`, `Heading(level)`, `Paragraph`, `Emphasis`, `Strong`, `Link`, `Code`, `CodeFence(lang, content)`, `MermaidBlock(source)`, `Table`, `ListBlock`, `ListItem`, `Image(src, alt)`, `BlockQuote`, `ThematicBreak`, `Html` (raw, optional pass-through)

AST transformers run in this order:

1. `strip_yaml_frontmatter` — remove leading `---…---`.
2. `normalize_merged_atx_headings` — split run-on `# Part## Chapter`.
3. `filter_metadata_blocks` — strip std-document bullets and Contributor Roles.
4. `promote_toc` — locate `## 目录` / `## Table of Contents` and reposition after H1.
5. `extract_mermaid_blocks` — convert `CodeFence(lang="mermaid")` → `MermaidBlock`.
6. `collect_outline` — build heading → bookmark map, clamping outline depth jumps to ≤ +1.

Each transformer is unit-testable (markdown → AST → expected AST).

#### 2.1.4 Asset Resolution Phase

All external assets resolved **concurrently** via `asyncio.gather` with caching:

- **Mermaid blocks** → `MermaidRenderer` chain:
  1. If `KROKI_URL` set → `mermaid_kroki.py` (HTTP POST to `/mermaid/svg` then svg→png via cairosvg).
  2. Elif Puppeteer extras installed → `mermaid_puppeteer.py` (mmdc subprocess with locked Chromium).
  3. Else → `mermaid_pure.py` (mermaid-py, lower fidelity, last resort).
  - Cache key: `sha256(source + theme + renderer_version)`.
  - Resource limits: ≤ 50K chars / ≤ 500 nodes / ≤ 10-deep subgraphs / 30s timeout / max 4 concurrent / output PNG ≤ 10MB.
  - Input lint runs **before** dispatch; rejects `click … callback`, raw `<script>`, `style=`, `img src=http(s)://`.

- **Image blocks** → `ImageRenderer`:
  - SVG → cairosvg → PNG.
  - Auto-downsample if ≥ 2400px to 300dpi.
  - Remote URLs (`http(s)://`) rejected unless `--allow-remote-assets` (off by default).

- **Code blocks** → `CodeRenderer` (Pygments tokenization, GitHub Light palette).

#### 2.1.5 Render Phase

`render/engine_base.py` dispatches to one of two engines based on the resolved template's `engine:` field.

**Engine A · ReportLab** (`render/engine_reportlab.py`) — default for narrative templates (`generic`, `user-manual`, `training`, `analysis-report`, `meeting-minutes`, `spec`, `whitepaper`, `proposal`):

| AST node | Flowable |
|---|---|
| `Heading` | `Paragraph` with heading style + outline entry |
| `Paragraph` | `Paragraph` (`wordWrap="CJK"` always set) |
| `Table` | `Table` with content-weighted column widths (`render/tables.py`) |
| `MermaidBlock` | `MermaidImage` Flowable (PNG + optional caption) |
| `CodeFence(lang)` | `FencedCodeCard` (lang badge + left accent + line numbers) |
| `BlockQuote` | `CalloutBox` (bordered, optional icon) |
| `Image` | `Image` Flowable with downsample-aware sizing |

Cover page (if template `cover.enabled: true`) rendered as a leading flowable group via the template's `cover/` section partials. Header/footer applied via `onFirstPage` / `onLaterPages` callbacks (locale-aware date formatting). `SimpleDocTemplate.build()` writes to a temp file (in tmpfs when available).

**Engine B · WeasyPrint** (`render/engine_weasyprint.py`) — default for highly-designed structured templates (`quote`, `certificate`, future `invoice`):

1. The template's HTML/CSS files (`template.html`, `style.css`, optional partials in `sections/`) are loaded.
2. The composer fills a Jinja2 sandbox context with: `front_matter` (validated structured fields), `body_html` (markdown body rendered to HTML by markdown-it-py), `brand` (theme tokens + asset URLs), `computed` (template-declared computed fields like `grand_total`).
3. Mermaid blocks are pre-rendered to inlined SVG/PNG (same renderer chain as Engine A; result inlined as `<img>` tags or SVG nodes).
4. Code blocks are pre-rendered to highlighted HTML by Pygments (HtmlFormatter).
5. WeasyPrint renders the assembled HTML+CSS document to PDF, with bookmarks driven by `<h1>`–`<h6>` and CSS `bookmark-level` properties.

**Engine selection rules:**

- The template's `engine:` field is authoritative.
- A template may declare `engine: chromium` (deferred to v2.x); engine selector raises `ENGINE_NOT_AVAILABLE` if extras not installed.
- Watermarks (§5), PDF/A (§2.1.6 step 4 — see also `core/pdfa/`), and other post-processing are **engine-agnostic** — they all run on the resulting PDF regardless of which engine produced it.

#### 2.1.6 Post-Process Phase

Sequential pikepdf/pypdf passes:

1. Overlay issuer card on last page.
2. Overlay footer (confidential string + page N/N) on all pages.
3. Apply watermark layers in order (each enabled per `policy.min_level` and request):
   - L1 visible diagonal (pypdf overlay)
   - L2 XMP metadata (pikepdf, includes HMAC field)
   - L3 steganographic (pikepdf inserts ~50 sub-pixel transparent text spans per page)
   - L4 encryption (if plugin loaded **and** PDF/A not requested — they are mutually exclusive)
   - L5 signature (if plugin loaded)
4. PDF/A-2b post-processing (if `--pdfa`):
   - Embed sRGB ICC profile.
   - Verify all fonts are subset-embedded.
   - Write XMP RDF.
   - Strip encryption (incompatible with PDF/A).
   - Optional: invoke veraPDF for conformance check.
5. Deterministic mode: rewrite all `/CreationDate` and `/ModDate` to `SOURCE_DATE_EPOCH`; render id derived from `sha256(input + brand + options)`.

#### 2.1.7 Output Phase

Atomic write: `output.pdf.tmp.<random>` → `fsync` → `rename`. Returns:

```python
@dataclass(frozen=True)
class RenderResult:
    output_path: Path
    render_id: str            # UUIDv4 (or deterministic hash in --deterministic mode)
    pages: int
    bytes: int
    sha256: str
    warnings: list[str]
    metrics: RenderMetrics    # parse_ms, render_ms, post_ms, total_ms
    bundle: dict[str, Path]   # extra files produced when --bundle was set
                              # e.g., {"json": "/path/out.json", "review_md": "/path/out.review.md"}
```

**Default output is the PDF only.** Optional bundle artifacts (opt-in via `--bundle json,review_md`):

- `out.json` — the final composed `RenderConfig` (brand id+version, template id+version, validated front-matter, computed fields, metrics, render_id). Useful for downstream review/approval workflows; never includes the markdown body content (only its sha256).
- `out.review.md` — a human-readable snapshot of the structured fields (e.g., "Quote No.: …", "Grand Total: …"). Designed to be diffable in PR review.

Bundle is **never enabled by default**, even for `quote` or `certificate` templates. Enabling it is a per-call decision (CLI `--bundle ...`, MCP `bundle: [...]`, API `bundle=[...]`).

`render.complete` audit log line written with output hash + size + bundle paths (if any).

### 2.2 Concurrency Model

- One `Pipeline.render()` call is single-threaded at the orchestration level.
- Asset resolution uses `asyncio` for I/O concurrency (Mermaid HTTP, image downloads).
- `Pipeline` instances are **safe to use across threads** if each thread has its own instance; the Pipeline holds no mutable shared state.
- A separate `mdpdf-server` process can host one Pipeline pool for concurrent requests.

### 2.3 Determinism Mode

When `--deterministic` is set or `SOURCE_DATE_EPOCH` env var is exported:

- All embedded timestamps frozen.
- Render id derived from `sha256(input_bytes + brand_id + brand_version + serialized_options)`.
- Mermaid PNG generation: only Kroki and Puppeteer renderers are deterministic-safe (Chromium pinned); mermaid-py pure renderer is rejected with `RENDERER_NON_DETERMINISTIC` error.
- Same input + same env produces bit-identical PDF (verified by `tests/golden/deterministic/*.sha256`).

### 2.4 Default Behaviors Summary

| Setting | Default | Override |
|---|---|---|
| Watermark levels | L1 + L2 + L3 | `--no-watermark` (only allowed if policy.min_level == L0) |
| L4 encryption | Off (interface only) | Requires plugin installed |
| L5 signature | Off (interface only) | Requires plugin installed |
| PDF/A | Off | `--pdfa` |
| Deterministic | Off | `--deterministic` or `SOURCE_DATE_EPOCH=…` |
| Audit log | On (user-level path `~/.md-to-pdf/audit.jsonl`) | `--no-audit` (rejected if policy enforces) |
| Remote assets | Rejected | `--allow-remote-assets` (rejected if policy forbids) |
| Mermaid renderer | Auto: Kroki → Puppeteer → pure | `--mermaid-renderer kroki|puppeteer|pure` |
| Locale | `en` | `--locale zh-CN` |
| CJK fonts | Not bundled (extras) | `pip install md-to-pdf[cjk-sc]` or `MD_PDF_FONT_DIR=…` |
| Template | `generic` | `--template <id>` or front-matter `template:` |
| Render engine | Decided by template | (template-controlled, not user-controlled) |
| Front-matter required-field missing | Warning, continue | `--strict` or `policy.template.strict_front_matter: true` |
| Output bundle | PDF only | `--bundle json,review_md` |
| Pre-release audit / business gating | **Not implemented** (out of scope; see §0.2) | Use external review systems |

### 3.5 Three-Layer Registry (Resolution Order)

### 4.3 Front-Matter Schema (rich, typed, with imports)

`front_matter_schema.yaml` defines what structured data the markdown front-matter must / may carry. Implementation uses pydantic v2 under the hood.

```yaml
schema_version: "1.0"

# External-file imports — merged into the front-matter at validate time.
# Paths are sandboxed: must be inside the markdown's working dir, the brand pack,
# the template pack, or a directory whitelisted in policy.assets.allowed_input_dirs.
imports:
  - field: pricing
    source: ./.pricing_config.yaml          # relative to markdown file
    required: true
    description: "Tenant-level pricing governance: rate card, discount, PM fee"
  - field: issuer
    source: ./.issuer.yaml                   # or company-profile shared file
    required: true
  - field: prefill
    source: ./.customer_prefill.yaml         # client-level pre-filled fields
    required: false

# Field definitions
fields:
  # Top-level metadata
  quotation_no:
    type: string
    required: true
    pattern: "^[A-Z]{2,5}-[A-Z]{2,5}-\\d{8}-\\d+$"
    description: "Format: ISSUER-TERRITORY-YYYYMMDD-SEQ"
  issue_date: {type: date, required: true}
  valid_until: {type: date, required: true}
  territory: {type: string, required: true, examples: ["APAC_HK", "GREATER_CHINA"]}
  currency: {type: enum, values: [HKD, CNY, USD, EUR, JPY, GBP], required: true}
  subject: {type: string, required: true, max_length: 200}

  # Issuer block (typically populated by import)
  issuer:
    type: object
    required: true
    fields:
      legal_entity: {type: string, required: true}
      company_number: {type: string, required: true}
      address: {type: string, required: true}
      email: {type: email, required: true}
      phone: {type: string, required: true}
      website: {type: url, required: true}
      acceptance_party_name: {type: string, required: true}
      representative:
        type: object
        fields:
          name: {type: string, required: true}
          title: {type: string, required: true}
          email: {type: email, required: true}
          phone: {type: string, required: true}

  # Client block
  client:
    type: object
    required: true
    fields:
      legal_name: {type: string, required: true}
      company_number: {type: string, required: false}
      registered_address: {type: string, required: true}
      website: {type: url, required: false}
      contact_name: {type: string, required: true}
      contact_email: {type: email, required: true}
      contact_phone: {type: string, required: true}
      acceptance_party_name: {type: string, required: true}

  # Line items array
  line_items:
    type: array
    required: true
    min_length: 1
    item_schema:
      category:
        type: enum
        values: [implementation_service, hardware, subscription,
                 cr_custom_development, professional_outsourcing_service,
                 payment_gateway_fee, project_management_fee]
        required: true
      description: {type: string, required: true}
      quantity: {type: decimal, required: true, min: 0}
      unit: {type: enum, values: [MD, hour, license, device, month, year], required: true}
      unit_price: {type: decimal, required: true, min: 0}
      billing_cycle: {type: enum, values: [one_time, monthly, quarterly, annual], required: true}
      discount_type: {type: enum, values: [none, percentage, absolute, volume_discount], default: none}
      discount_value: {type: decimal, default: 0, min: 0}
      price_source: {type: string, required: false}
      remarks: {type: string, required: false}

  # Payment terms (multiple sub-clauses)
  payment_terms:
    type: object
    required: true
    fields:
      invoice_trigger: {type: string, required: true}
      service_payment_term: {type: string, required: true}
      service_milestone_schedule: {type: string, required: false}
      hardware_payment_term: {type: string, required: false}
      subscription_payment_term: {type: string, required: false}
      payment_gateway_fee_term: {type: string, required: false}
      late_payment_clause: {type: string, required: true}

  # Legal clauses
  legal_clauses:
    type: object
    required: true
    fields:
      governing_law_clause: {type: string, required: true}
      limitation_of_liability_clause: {type: string, required: true}
      confidentiality_clause: {type: string, required: true}
      effectiveness_clause: {type: string, required: true}
      execution_counterparts_clause: {type: string, required: true}
      signed_copy_clause: {type: string, required: true}
      quotation_change_reservation_clause: {type: string, required: true}
      third_party_dependency_clause: {type: string, required: false}
      scope_boundary_note: {type: string, required: false}
      travel_note: {type: string, required: false}
      special_note: {type: string, required: false}

  # Bank info (multi-currency support)
  bank:
    type: object
    required: true
    fields:
      beneficiary_bank: {type: string, required: true}
      beneficiary_name: {type: string, required: true}
      bank_address: {type: string, required: true}
      bank_code: {type: string, required: false}
      swift_code: {type: string, required: true, pattern: "^[A-Z]{6}[A-Z0-9]{2}([A-Z0-9]{3})?$"}
      accounts:
        type: array
        item_schema:
          currency: {type: enum, values: [HKD, CNY, USD, EUR, JPY, GBP]}
          account_number: {type: string, required: true}

  # Tax
  tax:
    type: object
    required: true
    fields:
      applicability: {type: string, required: true}
      calculation_summary: {type: string, required: true}
      rate: {type: decimal, required: false}
      total: {type: decimal, required: false}
      note: {type: string, required: false}
      staleness_warning: {type: string, required: false}

# Computed fields — evaluated after merge + validation; available to template renderers
# Expression language: a small whitelisted subset of Python expressions evaluated by asteval
# (no imports, no attribute access on dunder names).
computed_fields:
  service_total:
    expr: "sum(item.quantity * item.unit_price * (1 - item.discount_value/100 if item.discount_type=='percentage' else 1) for item in line_items if item.category in ['implementation_service','cr_custom_development','professional_outsourcing_service'])"
    type: decimal
  hardware_total:
    expr: "sum(item.quantity * item.unit_price for item in line_items if item.category == 'hardware')"
    type: decimal
  subscription_total:
    expr: "sum(item.quantity * item.unit_price for item in line_items if item.category == 'subscription')"
    type: decimal
  pm_fee_total:
    expr: "service_total * pricing.project_management_fee.fee_pct_of_authorized_implementation_amount / 100 if pricing.project_management_fee.enabled else 0"
    type: decimal
  grand_total:
    expr: "service_total + hardware_total + subscription_total + pm_fee_total + (tax.total or 0)"
    type: decimal

# Cross-field validations — failures emit warnings (not errors) by default; --strict elevates to error
validations:
  - name: "valid_until_after_issue_date"
    expr: "valid_until > issue_date"
    message: "valid_until must be after issue_date"
    severity: error                     # always error (logical impossibility)

  - name: "currency_matches_pricing_config"
    expr: "currency == pricing.expected_currency if pricing else true"
    message: "Quote currency must match pricing config currency"
    severity: warning                   # may legitimately differ, but suspicious

  - name: "grand_total_non_negative"
    expr: "grand_total >= 0"
    message: "Grand total cannot be negative"
    severity: error

  - name: "line_items_present"
    expr: "len(line_items) >= 1"
    message: "Quote must include at least one line item"
    severity: error
```

**Behavior summary:**

- **Type errors and pattern errors** always fail the render (exit code 2 from CLI).
- **Required field missing** is a warning by default; `--strict` or `policy.template.strict_front_matter: true` escalates to error.
- **Cross-field validation** severity is declared per-rule (error / warning).
- **Computed fields** are available to renderers as if they were front-matter fields. They are also written to the bundle `out.json` if `--bundle json` is set.
- **Imports** are resolved in the order declared; later imports may reference fields from earlier ones.

### 4.4 Dual Rendering Engine

| Engine | Best for | Pros | Cons |
|---|---|---|---|
| `reportlab` | Narrative documents (manual, training, report, proposal, spec, whitepaper, meeting-minutes) | Mature CJK + watermark + bookmark support inherited from v1.8.9; pure Python; deterministic; no extra deps | Layout written in code; designer iteration slower; HTML/CSS knowledge does not transfer |
| `weasyprint` | Highly-designed structured documents (quote, certificate, future invoice / contract / certificate) | Designer-friendly HTML+CSS; print-CSS standard; pure Python (~30MB, no Chromium); good Unicode + bidi | Less mature CJK line breaking than ReportLab; no JavaScript (cannot render dynamic charts; pre-render via Mermaid pipeline instead) |
| `chromium` | Templates needing JS / advanced flexbox / web-only CSS features | Full browser fidelity | 500MB Chromium download; security review burden; **deferred to v2.x** |

**Engine-agnostic post-processing** — watermarks, PDF/A, audit logging, bundle generation all run on the produced PDF regardless of engine. This means a quote rendered by WeasyPrint gets the same L1+L2+L3 watermarks as a manual rendered by ReportLab.

**Migration path for the contributed quotation HTML template** (`examples/contributed/quotation/`):

1. Move `quotation_template_zh_hk_v1.0.html` → `template/builtin/quote/template.html` (with minor Jinja2 variable renames).
2. Extract front-matter schema from the JSON manifest (already done in §4.3 — the field list above is sourced from your real TJI quote).
3. Move CSS into `template/builtin/quote/style.css`.
4. Replace Playwright-Chromium rendering with WeasyPrint (most CSS portable; print-CSS rules will need light adjustment).
5. Existing locale variants (`zh_HK`, `zh_CN`) become `template/builtin/quote/locales/zh-HK/` and `locales/zh-CN/`.

### 4.6 Composition with Brand

`BrandTemplateComposer` resolves the effective render config:

| Field group | Source of truth |
|---|---|
| Identity (logo, brand colours, fonts, footer text, issuer) | **Brand wins** |
| Layout (margins, page size, header per-page, signature block, TOC presence) | **Template wins** (template.layout overrides brand.layout) |

- Brand declares `layout.show_toc: true`, template `quote` declares `layout.show_toc: false` → effective `false` (template wins).
- Brand declares `security.watermark_min_level: L1`, template `certificate` declares `L3` → effective `L3` (strictest).
- Brand `acme` has `security.allow_remote_assets: false`, template `proposal` declares `allow_remote_assets: true` → effective `false` (security strictest wins).

### 4.8 Output Bundle (Opt-In Only)

Default output is the PDF only. Bundle artifacts are produced **only when explicitly requested**:

| Artifact | Content | Use case |
|---|---|---|
| `out.json` | RenderConfig snapshot: brand id+version, template id+version, validated front-matter, computed fields, metrics, render_id; markdown body referenced only by sha256 | Downstream review / approval workflows; SIEM / DLP integration; future legacy-system replay |
| `out.review.md` | Human-readable structured-field snapshot ("Quote No.: …", "Grand Total: …", "Valid Until: …") | PR review diffability; quick sanity check before sending |

A template's `bundle_capabilities:` declares which artifacts it knows how to produce; a render call's `bundle:` parameter selects which to produce. If `bundle:` requests an artifact not in `bundle_capabilities`, the engine raises `BUNDLE_NOT_SUPPORTED`.

The `certificate` template hard-disables bundle (sets `bundle_disallowed: true`) — certificates by nature should not have a separate JSON snapshot of their content.

### 4.9 Schema Compatibility Promise

Identical model to brand pack schema:

- `schema_version` follows SemVer.
- **Patch / minor:** old template packs are guaranteed to work.
- **Major:** `md-to-pdf template migrate <path>` migrates in place; runtime auto-upgrades for one major cycle with deprecation warning before hard-failing.
- Built-in templates are **part of the core wheel** and version-locked to the core version. Users who need a stable template independent of core version should copy the built-in into `~/.md-to-pdf/templates/<id>/` and pin it.

### 4.10 What Is Explicitly NOT in the Template System

To keep template scope sharply bounded:

- **No pre-release audit / business gating.** Templates declare structural validation (required fields, types, cross-field consistency) but **not** business approval workflows. The user's existing `pre_release_audit.verdict` field is a downstream concern outside this product. (See §0.2 Non-Goals.)
- **No LLM-assisted wording adjustment.** The `wording_assist` mode in the contributed reference is out of v2.0 scope. Future v2.x may add a separate `mdpdf-assist` companion package; not core.
- **No commercial-process signature gating.** L5 digital signature exists for cryptographic non-repudiation, not for "manager has approved this quote" workflow. The latter is a CRM concern.
- **No "template structure guard"** as a distinct downstream gate — but the template pack itself is schema-validated at load time, equivalent in effect.

## 5 · Security & Watermark Stack

### 5.1 Threat Model (Honest Scope)

**In scope (we defend):**

| Threat | Mitigation |
|---|---|
| Screenshot leak | L1 visible diagonal watermark with user/time/render-id |
| File forwarding loses provenance | L2 XMP metadata (mdpdf:RenderId, mdpdf:RenderUser, …) |
| User crops out visible watermark | L3 steganographic (sub-pixel transparent text, ~50/page) |
| Unauthorized PDF open | L4 encryption (interface; pluggable) |
| PDF tampering / forgery | L5 digital signature (interface; pluggable, plus L2 HMAC) |
| Mermaid source code injection (XSS via PDF) | AST lint + Kroki/Puppeteer process isolation |
| Brand field tampering removes footer | `forbidden_override_fields` schema enforcement |
| Internal user uses different brand to impersonate | `policy.yaml` brand-lock |
| Path traversal via brand assets | `safe_join` validation; reject `../`, `file://`, absolute paths outside brand root |
| Brand secretly loads remote tracker image | `allow_remote_assets: false` default |
| Crash leaves sensitive temp files on disk | tmpfs + context-manager cleanup + `atexit` hook |

**Out of scope (we do not promise):**

- Manual transcription from screen (no watermark defends against re-typing)
- OS-level screen capture tools (EDR/DLP territory)
- Format conversion that strips watermarks (e.g., re-OCR + new PDF) — L2 lost, L3 partially lost
- Physical photography of the screen (L1 deters; nothing prevents)
- Sophisticated image-processing attacks against L3 (we estimate ~70% success on print-then-scan)

### 5.2 L1 — Visible Diagonal Watermark

- Mechanism: pypdf overlay on every page.
- Style: 13pt, 38° rotation, 120pt row spacing, light gray RGB(0.92, 0.93, 0.94).
- Content: template string `{brand_name} // {user} // {render_date}` (brand-customizable).
- Controls: `--watermark-user`, `--watermark-text`, `--no-watermark` (only valid if `policy.min_level == L0`).

### 5.3 L2 — XMP Metadata Watermark

Pikepdf writes XMP RDF with a custom `mdpdf:` namespace alongside Dublin Core:

| XMP Key | Source |
|---|---|
| `dc:creator` | brand_name |
| `dc:title` | document H1 or brand default |
| `pdf:Producer` | `md-to-pdf 2.x` |
| `xmp:CreatorTool` | `md-to-pdf 2.x` |
| `xmp:CreateDate` | ISO 8601 (frozen in deterministic mode) |
| `mdpdf:RenderId` | UUIDv4 (or derived hash in deterministic mode) |
| `mdpdf:RenderUser` | request.watermark.user |
| `mdpdf:RenderHost` | `sha256(hostname)[:16]` (no plaintext leak) |
| `mdpdf:BrandId` | brand id |
| `mdpdf:BrandVersion` | brand pack version |
| `mdpdf:InputHash` | `sha256(markdown_bytes)` |
| `mdpdf:WatermarkLevel` | e.g., `"L1+L2+L3"` |
| `mdpdf:Hmac` | `HMAC-SHA256(all-mdpdf-fields-canonicalized, secret)` |

`secret` is loaded from `MD_PDF_HMAC_SECRET` env var or `policy.yaml > audit > hmac_secret`. If absent, the HMAC field is omitted (degraded mode). Verification tool: `md-to-pdf forensics verify <pdf>`.

### 5.4 L3 — Steganographic Watermark

- Mechanism: pikepdf inserts ~50 transparent text spans per page distributed in low-density regions.
- Encoding: render-id + user-hash repeated with Reed-Solomon-like error correction (we use a simpler repetition code for v2.0; can upgrade later).
- Style: 0.5pt font size, alpha 0.02, color matching background within ΔE < 2.
- Robustness:
  - Crop ≤ 80% page → recoverable
  - JPG re-save (lossy) → recoverable
  - Print + scan → ~70% recovery
  - OCR + regenerate PDF → completely lost (acknowledged limitation)
- Cost: PDF size +1–3%, render time +50ms/page.
- Extraction tool: `md-to-pdf forensics extract <pdf>` returns recovered render-id with confidence score.

### 5.5 L4 — Encryption (Plugin Interface)

```python
# core/security/encryption.py
class EncryptionPlugin(ABC):
    @abstractmethod
    def apply(self, pdf_path: Path, options: EncryptionOptions) -> Path: ...

class NoOpEncryption(EncryptionPlugin):
    def apply(self, pdf, opts):
        if opts.required:
            raise EncryptionError("L4 plugin not loaded")
        return pdf
```

- Discovery: Python entry point group `mdpdf.encryption_plugins`.
- Default: `NoOpEncryption`. v2.0 ships no concrete encryption; we publish reference plugin `md-to-pdf-encryption-pikepdf` separately in v2.1.
- Mutually exclusive with PDF/A (PDF/A-2b forbids encryption).

### 5.6 L5 — Digital Signature (Plugin Interface)

```python
class SignaturePlugin(ABC):
    @abstractmethod
    def sign(self, pdf_path: Path, options: SignOptions) -> Path: ...
```

- Discovery: entry point group `mdpdf.signature_plugins`.
- Default: NoOp.
- Reference plugin (community-maintained, not in core): `md-to-pdf-signature-endesive` (PAdES via endesive).
- Enterprises typically wire their PKI/HSM/cloud CA into a custom plugin.

### 5.7 Audit Log

**Format:** JSONL, two events per render (`render.start`, `render.complete` or `render.error`).

**Default location:** `~/.md-to-pdf/audit.jsonl` (user-mode) or `/var/log/md-to-pdf/audit.jsonl` (system-mode via policy).

**Rotation:** daily, retain 90 days by default (`policy.audit.retain_days`).

**Privacy:** input source is **never** copied; only `sha256` recorded. Output PDF likewise referenced by hash + path + size.

**Schema:** see appendix A.

**Permissions:** `0640`, owner + `audit` group.

**Forwarding:** optional syslog / SIEM forwarding via `policy.audit.forward_syslog`.

### 5.8 Mermaid Sandbox

**Input lint** (before dispatch):

- AST validator rejects: `click … callback`, raw HTML `<script>`, `style=`, `img src=http(s)://`.
- Length ≤ 50K chars (policy-tunable).
- Node count ≤ 500 (policy-tunable).
- Subgraph nesting depth ≤ 10.

**Process isolation by renderer:**

- **Kroki** — full HTTP isolation; Kroki container provides its own sandbox.
- **Puppeteer** — `--disable-features=NetworkService,Extensions`; `file://` denied; 30s timeout; subprocess `prlimit` RAM cap 512MB on Linux.
- **pure-Python** — no browser, no subprocess; safest by construction.

**Output limits:**

- Max output PNG ≤ 10MB (defends against svg-bomb attacks).
- Max concurrent Mermaid renders ≤ 4.

### 5.9 Temp File & Cleanup Policy

- All temp files under `tmpfs` (`/dev/shm` on Linux, `/tmp` on macOS) when available.
- `core/cache/TempContext` context manager guarantees cleanup on success **and** exception.
- `atexit` hook scans for orphaned `mdpdf-tmp-*` directories on process exit.
- PDF output uses atomic write: `output.pdf.tmp.<random>` → `fsync` → `rename`.

**Exit codes:**

| Code | Meaning |
|---|---|
| 0 | Success |
| 1 | Render failure (see stderr) |

### 6.3 Skill Bundle

`packages/skill/SKILL.md` is a **thin wrapper** that teaches the agent how to invoke the CLI. It does **not** ship Python code. Prerequisite: `pip install md-to-pdf`.

This means:
- Upgrading the CLI upgrades skill behavior with no skill redistribution.
- Skill bundle is ~5KB.
- One skill bundle works for Claude Code, Cursor, Gemini CLI, and any future SKILL.md-compatible client.

**Template guidance in SKILL.md** — the skill explicitly tells the agent:
- **Always declare a template explicitly.** The skill must not infer business intent. Either pass `--template <id>` or include `template: <id>` in markdown front-matter. If the user's intent is ambiguous (e.g., "make this into a quote" — but no `quote` template fields are present), ask the user before rendering.
- **Use `md-to-pdf template list` to discover available templates.** Don't assume a template id from training data; the user's project may have custom templates in `./.md-to-pdf/templates/` that override built-ins.
- **Templates with structured front-matter** (e.g., `quote`, `certificate`) usually need extra import files (e.g., `--prefill`, `--issuer`). The skill provides example invocations for the 10 built-in templates.
- **For unfamiliar templates**, run `md-to-pdf template show <id>` first to see the required front-matter fields and required imports.

## 7 · Testing, CI, Observability, Versioning, Release

### 7.1 Test Pyramid

- **Unit (~500 tests)** — every module independently. AST transformers, brand schema, policy, safe_paths, watermark layers each get their own test file.
- **Integration (~100 tests)** — full `Pipeline.render()` with real renderers (Kroki via dockerized fixture).

- **Stress** — 1000-page docs (RAM + time bounds), 100 Mermaid blocks (concurrency limits), 4K images (downsampling).

### 7.2 Golden Testing (Four Layers + Combinatorics)

Direct PDF byte diff is unstable due to embedded timestamps and ZLIB compression. Instead:

| Layer | What it catches | Update cost |
|---|---|---|
| AST diff | Parse changes (most stable) | Low |
| XMP diff | Post-process metadata | Low |
| Text-layer diff (`pdftotext` extract) | Content regressions | Low |
| Layout fingerprint (page count + per-page text-box count) | Layout regressions | Low |
| sha256 (deterministic mode only) | CI archival guarantee | Highest |

`tests/golden/` directory structure:

```

├── templates/                          # one fixture per built-in template
│   ├── generic/         {input.md, expected.ast.json, expected.xmp.json, expected.text.txt, expected.fingerprint.json}
```

### 7.6 Versioning Policy (SemVer)

- **MAJOR** — breaks `core` public API, brand schema, CLI exit codes, or MCP tool schemas.
- **MINOR** — additive features, new renderers, new brand fields (backwards-compatible).
- **PATCH** — bug + security fixes only.

`mdpdf-core`, `mdpdf-cli`, `mdpdf-mcp` ship in **lockstep** on the same version number for cognitive simplicity.

**Stable surfaces:**
- `core` public API (Pipeline, RenderRequest, RenderResult, MdpdfError subclasses).
- Brand pack schema (within MAJOR).
- Template pack schema and front-matter schema language (within MAJOR).
- Built-in template ids and their `front_matter_schema` field names (within MAJOR — only additive changes).
- CLI subcommands and primary flags.
- MCP tool schemas (additive within MAJOR).

**Not stable (subject to change in MINOR):**
- Internal modules (`core.markdown.transformers.*`, `core.render.flowables.*`).
- PDF byte-level output (except in deterministic mode within same MINOR).
- Log event names (consume by prefix, not exact match).

**Deprecation flow:** introduce in vN.x → warn for one minor cycle → remove in v(N+1).0 with migration guide.

## 8 · Migration from v1.8.9

### 8.1 What Carries Over

- ReportLab + pypdf + Pygments + mermaid-cli core stack proven; we keep these.
- CJK Noto Sans SC handling logic (extract from monolith into `core/fonts/`).
- Brand pack concept (existing `brand_kits/` becomes a v1 brand; provide migration command to v2).
- Mermaid CSS injection + viewport presets (preserve as Puppeteer renderer behavior).
- Watermark visible diagonal (becomes L1 implementation).
- TOC promotion + bookmark collection (becomes AST transformer).
- Heading-merge split (becomes AST transformer).
- Pygments GitHub Light palette (preserve in `core/renderers/code_pygments.py`).
- Contributed quotation HTML template (`examples/contributed/quotation/`) — its data schema (~30 fields across issuer / client / line items / payment terms / legal clauses / bank info) becomes the authoritative reference for the built-in `quote` template's `front_matter_schema.yaml`. Its HTML/CSS becomes `template/builtin/quote/template.html` + `style.css` (Playwright → WeasyPrint port).

### 8.2 What Is Removed

- Hardcoded `IDIMSUM` company references.
- Hardcoded `/company_assets/brand/...` path fallbacks.
- macOS browser fallback chain (replaced by single Puppeteer-locked Chromium when that renderer is selected).
- Single global `brand_kits/` directory at repo root (replaced by 3-layer registry).
- regex-based markdown parsing (replaced by markdown-it-py AST).
- 3,443-line monolith `scripts/md_to_pdf.py` (replaced by modular `packages/core/`).
- `pre_release_audit` workflow gate from the legacy quotation system — out of scope (see §0.2 and §4.10). Templates only validate structure, not business approval.
- `wording_assist` LLM rewrite mode — out of scope for v2.0; may become a separate `mdpdf-assist` companion package later.

### 8.3 Migration Tooling

- `md-to-pdf brand migrate <path>` — converts v1 brand pack (theme.yaml + compliance.md) to v2 (brand.yaml + theme.yaml + compliance.yaml).
- `md-to-pdf template migrate <path>` — converts v0 template packs (legacy HTML/Python templates) to v2 template-pack layout; for the contributed quotation template, automates the Playwright → WeasyPrint CSS adjustment.
- `md-to-pdf doctor` — analyses environment, reports missing fonts / Mermaid renderer / policy issues with remediation steps.

### 8.4 v1 → v2 Compatibility Window

- v2.0 ships with `--legacy-brand` flag accepting v1 layout.
- v2.0 documents that v1 markdown documents (no template declared) automatically resolve to the `generic` template — they render with v1.8.9-equivalent behaviour.
- v2.1 adds deprecation warning when `--legacy-brand` used.
- v3.0 removes `--legacy-brand`.

## 9 · Open Decisions Deferred to Implementation

These are decided in spirit but specific values to be set in the implementation plan:

- **GitHub organisation name** — used throughout this doc as `your-org` (e.g., `ghcr.io/your-org/md-to-pdf`, `your-org/md-to-pdf-action@v2`); to be replaced once the public repository name is chosen.
- **Project package name on PyPI** — `md-to-pdf` is preferred but may be taken; `mdpdf` is the fallback.
- Exact `KROKI_URL` health-check timeout (likely 2s).
- Default `MD_PDF_FONT_DIR` resolution order on Windows.
- Specific Pygments style customisations beyond GitHub Light.
- Whether `forensics extract` requires the same HMAC secret used at render time (recommended: yes).
- Specific veraPDF Docker image to vendor for PDF/A validation.
- OpenTelemetry exporter — targeted for v2.2 or v2.3; not v2.0.
- **Computed-field expression engine** — `asteval` is the working choice for safe Python-subset evaluation; alternatives (`simpleeval`, custom DSL) to be benchmarked during implementation for performance and security audit ease.

Acceptance threshold: must produce visually equivalent line breaks for the existing CJK fixtures (`fixtures/uat-cjk.md`, `fixtures/uat-zh.md`).
- **Chromium engine** — deferred to v2.x; revisit if any v2.0 template proves WeasyPrint cannot render acceptably.
- **Font policy for `weasyprint` engine** — WeasyPrint reads `@font-face` from CSS; need to confirm the brand-pack font dir is reachable and respected. Likely require an explicit URL prefix or copy-into-tmp step at render start.

These do not block implementation; they are tuning decisions made when each subsystem lands.
