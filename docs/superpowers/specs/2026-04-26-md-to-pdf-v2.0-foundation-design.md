## 0 ┬╖ Goal & Non-Goals

### 0.1 Goal

Refactor the v1.8.9 markdown-to-PDF monolith into a **modular, testable, open-source foundation** that future versions can extend without reorganising the codebase. v2.0 is the structural floor; it does not aim for feature growth.

The four enterprise pillars from the long-term vision still guide v2.0 тАФ but each is partially realised:

1. **Reliability** тАФ deterministic mode, version-locked dependencies, 4-layer golden testing, structured errors, atomic file writes.
2. **Compatibility** тАФ single core engine consumable by CLI and Python API; multi-platform OS matrix; SemVer compatibility guarantees. (MCP / GitHub Action / Skill adapters land in v2.2.)
3. **Security** тАФ L1 visible + L2 XMP watermarks; brand schema validation; path sandboxing; audit log; safe Mermaid handling. (L3 steganographic, L4 encryption, L5 signature land in v2.3.)
4. **Future extensibility** тАФ the architecture exposes a `RenderEngine` ABC with one concrete implementation (ReportLab), plus a `Renderer` ABC for Mermaid/image/code, and a brand-pack v2 schema with a SemVer compatibility promise. (Template-pack system, second engine, plugin interfaces follow in v2.1+.)

**v2.0 single acceptance bar:** every existing v1.8.9 fixture (`fixtures/uat-*.md`, `fixtures/fenced-mermaid-smoke.md`, `fixtures/mermaid-noto-presets.md`) **plus** the new comprehensive fixture `fixtures/branch_ops_ai_robot_product_brief.md` (see ┬з7.2) produces visually-equivalent output via the new pipeline, validated against current production PDFs. The new fixture is the dominant gate тАФ it exercises every renderer code path in one document, so a single visual regression there usually surfaces issues that the small focused fixtures miss.

### 0.2 Non-Goals (v2.0)

- **Not** a Word/HTML/EPUB exporter.
- **Not** a markdown editor / live preview tool.
- **Not** a DRM platform.
- **No template-pack system** тАФ single `generic` template (= v1.8.9 default behaviour). Template system is v2.1.
- **No second rendering engine** тАФ ReportLab only. WeasyPrint considered and explicitly deferred to v2.x, only if real users ask for HTML-authored templates (rationale: one engine = one test matrix, one CJK story, one determinism story; the existing v1.8.9 ReportLab pipeline already meets the user's quote/certificate needs).
- **No MCP / GitHub Action / Skill bundle / Docker images** тАФ CLI + Python API only. All four adapters land in v2.2.
- **No L3-L5 watermarks** тАФ L1 visible + L2 XMP only. L3 steganographic deferred to v2.3 (requires a working prototype before threat-model claims).
- **No `policy.yaml` system** тАФ brand-level `security:` settings only. System-wide policy enforcement and brand-lock land in v2.3.
- **No PDF/A-2b mode**, **no forensics extract/verify tools**, **no veraPDF in CI**, **no Prometheus metrics**, **no OpenTelemetry** тАФ all v2.3 or later.
- **No commercial-process gating** (pre-release audit, wording-assist) тАФ permanently out of scope.

## 2 ┬╖ Rendering Pipeline (Data Flow)

### 2.1 Phases

```
Input тЖТ Validate тЖТ Parse тЖТ Asset Resolution тЖТ Render тЖТ Post-Process тЖТ Output
```

#### 2.1.1 Input Phase

`RenderRequest` is the unified input contract for CLI and Python API:

```python
@dataclass(frozen=True)
class RenderRequest:
    source: str | Path                  # markdown content or file path
    source_type: Literal["content", "path"]
    output: Path
    brand: str | InlineBrand | None     # name or dict
    brand_overrides: dict[str, Any]     # field-level overrides (whitelist enforced)
    template: str = "generic"           # only "generic" valid in v2.0; field exists
                                        # for forward compatibility with v2.1 templates
    watermark: WatermarkOptions         # user, level (L0|L1|L1+L2), custom_text
    deterministic: bool                 # SOURCE_DATE_EPOCH-respecting
    locale: str                         # zh-CN / en
    audit_enabled: bool                 # default True
```

CLI arguments and markdown front-matter merge with **CLI > front-matter > brand defaults** precedence. Markdown front-matter may declare `template:` and `brand:`; both are validated against allowlists during the validate phase below.

#### 2.1.2 Validate Phase (fail fast on structure)

Order matters; each step can short-circuit with a structured error:

1. `brand/registry`: resolve brand id through 3-layer overlay or accept inline brand.
2. `brand/schema`: pydantic v2 validation of resolved brand pack.
3. `brand/safe_paths`: validate brand asset paths (no `../`, no `file://`, no absolute paths outside brand root).
4. Template allowlist check: `request.template` (or front-matter `template:`) must be `"generic"`; any other value raises `TEMPLATE_NOT_FOUND` with a message pointing to the v2.1 release notes. (The check is centralised here so v2.1 can replace the allowlist with a registry lookup without changing call sites.)
5. `fonts/manager`: confirm CJK fonts available iff input markdown contains CJK chars (byte-level regex on raw input тАФ no parsing required); else fail loudly with `FONT_NOT_INSTALLED`.
6. `security/audit`: emit `render.start` log line with input hash, brand, template, render id (UUIDv4).

#### 2.1.3 Parse Phase

`markdown-it-py` produces tokens тЖТ converted to internal AST in `markdown/ast.py`. AST node types:

- `Document`, `Heading(level)`, `Paragraph`, `Emphasis`, `Strong`, `Link`, `Code`, `CodeFence(lang, content)`, `MermaidBlock(source)`, `Table`, `ListBlock`, `ListItem`, `Image(src, alt)`, `BlockQuote`, `ThematicBreak`, `Html` (raw, optional pass-through)

AST transformers run in this order:

1. `strip_yaml_frontmatter` тАФ remove leading `---тАж---`.
2. `normalize_merged_atx_headings` тАФ split run-on `# Part## Chapter`.
3. `filter_metadata_blocks` тАФ strip standardised document metadata bullets and Contributor Roles (preserves v1.8.9 default-strip behaviour).
4. `promote_toc` тАФ locate `## чЫох╜Х` / `## Table of Contents` and normalise.
5. `extract_mermaid_blocks` тАФ convert `CodeFence(lang="mermaid")` тЖТ `MermaidBlock`.
6. `collect_outline` тАФ build heading тЖТ bookmark map, clamping outline depth jumps to тЙд +1.

Each transformer is unit-testable (markdown тЖТ AST тЖТ expected AST).

**markdown-it-py parity is the highest-risk task in v2.0.** First implementation milestone is to run every existing v1.8.9 fixture through the new parser and produce a diff report against the regex-parser AST. Each diff is then triaged: accept (genuine improvement), normalise (add a transformer to match v1 behaviour), or fix (parser config change). No v2.0 release without diff zero on existing fixtures.

#### 2.1.4 Asset Resolution Phase

All external assets resolved **concurrently** via `asyncio.gather` with caching:

- **Mermaid blocks** тЖТ `MermaidRenderer` chain:
  1. If `KROKI_URL` set тЖТ `mermaid_kroki.py` (HTTP POST to `/mermaid/svg` then svgтЖТpng via cairosvg).
  2. Elif Puppeteer extras installed тЖТ `mermaid_puppeteer.py` (mmdc subprocess with locked Chromium).
  3. Else тЖТ `mermaid_pure.py` (mermaid-py, lower fidelity, last resort).
  - Cache key: `sha256(source + theme + renderer_version)`.
  - Resource limits: тЙд 50K chars / тЙд 500 nodes / тЙд 10-deep subgraphs / 30s timeout / max 4 concurrent / output PNG тЙд 10MB.
  - Input lint runs **before** dispatch; rejects `click тАж callback`, raw `<script>`, `style=`, `img src=http(s)://`.
  - In `--deterministic` mode, only Kroki and Puppeteer are accepted; `pure` is rejected with `RENDERER_NON_DETERMINISTIC`.

- **Image blocks** тЖТ `ImageRenderer`:
  - SVG тЖТ cairosvg тЖТ PNG.
  - Auto-downsample if тЙе 2400px to 300dpi.
  - Remote URLs (`http(s)://`) rejected unless `--allow-remote-assets` (off by default).

- **Code blocks** тЖТ `CodeRenderer` (Pygments tokenisation, GitHub Light palette).

#### 2.1.5 Render Phase (ReportLab)

`render/engine_reportlab.py` consumes the AST and produces a list of ReportLab Flowables:

| AST node | Flowable |
|---|---|
| `Heading` | `Paragraph` with heading style + outline entry |
| `Paragraph` | `Paragraph` (`wordWrap="CJK"` always set) |
| `Table` | `Table` with content-weighted column widths (`render/tables.py`) |
| `MermaidBlock` | `MermaidImage` Flowable (PNG + optional caption) |
| `CodeFence(lang)` | `FencedCodeCard` (lang badge + left accent + line numbers; behaviour preserved from v1.8.9) |
| `BlockQuote` | `CalloutBox` (bordered, optional icon) |
| `Image` | `Image` Flowable with downsample-aware sizing |

Header/footer applied via `onFirstPage` / `onLaterPages` callbacks (locale-aware date formatting). `SimpleDocTemplate.build()` writes to a temp file (in tmpfs when available).

Engine selection logic exists but trivially returns `engine_reportlab` in v2.0. The dispatch path is exercised by tests so adding a second engine in v2.x is mechanical.

#### 2.1.6 Post-Process Phase

Sequential pikepdf/pypdf passes:

1. Overlay issuer card on last page (existing v1.8.9 behaviour).
2. Overlay footer (confidential string + page N/N) on all pages.
3. Apply L1 visible diagonal watermark (pypdf overlay, controllable per request).
4. Apply L2 XMP metadata watermark (pikepdf тАФ see ┬з5.3 for fields).
5. Deterministic mode: rewrite all `/CreationDate` and `/ModDate` to `SOURCE_DATE_EPOCH`; render id derived from `sha256(input + brand + serialised options)`.

#### 2.1.7 Output Phase

Atomic write: `output.pdf.tmp.<random>` тЖТ `fsync` тЖТ `rename`. Returns:

```python
@dataclass(frozen=True)
class RenderResult:
    output_path: Path
    render_id: str            # UUIDv4 (or deterministic hash in --deterministic mode)
    pages: int
    bytes: int
    sha256: str
    warnings: list[str]
    metrics: RenderMetrics    # parse_ms, render_ms, post_ms, total_ms

@dataclass(frozen=True)
class RenderMetrics:
    parse_ms: int
    asset_resolve_ms: int
    render_ms: int
    post_process_ms: int
    total_ms: int
```

`render.complete` audit log line written with output hash + size.

### 2.2 Concurrency Model

- One `Pipeline.render()` call is single-threaded at the orchestration level.
- Asset resolution uses `asyncio` for I/O concurrency (Mermaid HTTP, image downloads).
- `Pipeline` instances are **safe to use across threads** if each thread has its own instance; the Pipeline holds no mutable shared state.

### 2.3 Determinism Mode

When `--deterministic` is set or `SOURCE_DATE_EPOCH` env var is exported:

- All embedded timestamps frozen.
- Render id derived from `sha256(input_bytes + brand_id + brand_version + serialised_options + watermark_user)`.
- Mermaid: only Kroki and Puppeteer renderers are deterministic-safe (Chromium pinned); `pure` rejected with `RENDERER_NON_DETERMINISTIC`.
- Same input + same env + same user produces bit-identical PDF (verified by `tests/golden/deterministic/*.sha256`).

**Determinism contract is per-(input, brand, options, watermark_user)** тАФ different users produce different PDFs because the L1 watermark contains `{user}`. This is documented behaviour, not a bug.

### 2.4 Default Behaviours Summary

| Setting | Default | Override |
|---|---|---|
| Watermark levels | L1 + L2 | `--no-watermark` (only when brand `security.watermark_min_level` permits L0) |
| Deterministic | Off | `--deterministic` or `SOURCE_DATE_EPOCH=тАж` |
| Audit log | On (`~/.md-to-pdf/audit.jsonl`) | `--no-audit` |
| Remote assets | Rejected | `--allow-remote-assets` |
| Mermaid renderer | Auto: Kroki тЖТ Puppeteer тЖТ pure | `--mermaid-renderer kroki\|puppeteer\|pure` |
| Locale | `en` | `--locale zh-CN` |
| CJK fonts | Not bundled (extras) | `pip install md-to-pdf[cjk-sc]` or `MD_PDF_FONT_DIR=тАж` |

### 3.5 Three-Layer Registry (Resolution Order)

Highest priority wins; brand packs at different layers do **not** merge тАФ they replace.

| Order (1 = winner) | Path | Use Case |
|---|---|---|
| 1 | CLI `--brand-pack-dir <path>` | Explicit override |
| 2 | `./.md-to-pdf/brands/<id>/` | Project-local |
| 3 | `~/.md-to-pdf/brands/<id>/` | User personal |
| 4 | `<repo>/examples/brands/<id>/` | Built-in samples (test/demo only) |

(System-wide `/etc/md-to-pdf/brands/<id>/` lookup ships in v2.3 alongside `policy.yaml`.)

### 3.6 Inline Brand Mode

```bash
# Full inline yaml
md-to-pdf in.md -o out.pdf --brand-config ./tmp-brand.yaml

# Field overrides on existing brand (whitelist enforced)
md-to-pdf in.md -o out.pdf --brand acme \
  --override theme.colors.accent=#FF0000 \
  --override compliance.footer.subtitle="DRAFT"
```

`forbidden_override_fields` always rejects; `allowed_override_fields` (when defined) acts as a whitelist; if `allows_inline_override: false`, all overrides rejected.

### 3.7 Schema Compatibility Promise

- `schema_version` follows SemVer.
- **Patch / minor:** old brand packs are guaranteed to work.
- **Major:** `md-to-pdf brand migrate <path>` migrates in place; runtime auto-upgrades for one major cycle with a deprecation warning before hard-failing.

### 3.8 v1 тЖТ v2 Brand Migration

`md-to-pdf brand migrate <path>` converts the existing `brand_kits/` (v1: theme.yaml + compliance.md + flat assets) into v2 (brand.yaml + theme.yaml + compliance.yaml + assets/ subdir). The current repo's `brand_kits/` is treated as the canonical v1 input for migration testing.

`--legacy-brand` flag accepts the v1 layout at runtime so existing users are not forced to migrate immediately. Deprecation warning added in v2.1; removed in v3.0.

## 4 ┬╖ Template Placeholder

v2.0 implements **one** template, `generic`, which is hard-coded тАФ there is no template-pack file format yet. `generic` reproduces v1.8.9's default rendering behaviour: no cover page, no required front-matter fields, no schema validation, content rendered as-is.

The `template` field in `RenderRequest` exists and accepts only `"generic"` in v2.0; passing any other value raises `TEMPLATE_NOT_FOUND` with a message pointing to the v2.1 release notes.

The full template-pack system (typed front-matter schema, computed fields, cross-field validations, 3-layer registry, built-in templates beyond `generic`) is specified in ┬з4 of the v2.x roadmap document.

## 5 ┬╖ Security & Watermark Stack

(L3 steganographic, L4 encryption interface, L5 signature interface land in v2.3. v2.0 implements L1 + L2 + audit log + path sandbox + Mermaid sandbox.)

### 5.1 Threat Model (v2.0 Honest Scope)

**In scope (we defend):**

| Threat | Mitigation |
|---|---|
| Screenshot leak | L1 visible diagonal watermark with user/time/render-id |
| File forwarding loses provenance | L2 XMP metadata (mdpdf:RenderId, mdpdf:RenderUser, тАж) |
| Mermaid source code injection | AST lint + Kroki/Puppeteer process isolation |
| Brand field tampering removes footer | `forbidden_override_fields` schema enforcement |
| Path traversal via brand assets | `safe_join` validation; reject `../`, `file://`, absolute paths outside brand root |
| Brand secretly loads remote tracker image | `allow_remote_assets: false` default |
| Crash leaves sensitive temp files on disk | tmpfs + context-manager cleanup + `atexit` hook |

**Out of scope for v2.0 (defended in later versions):**

- Cropping out the visible watermark тАФ needs L3 steganographic (v2.3).
- Unauthorised PDF open тАФ needs L4 encryption (v2.3).
- PDF tampering / forgery тАФ needs L5 signature + L2 HMAC (v2.3).

**Permanently out of scope (no version will defend):**

- Manual transcription from screen.
- OS-level screen capture tools.
- OCR + new PDF (XMP and any future steg watermark are lost).
- Physical photography of the screen.

### 5.2 L1 тАФ Visible Diagonal Watermark

- Mechanism: pypdf overlay on every page.
- Style: 13pt, 38┬░ rotation, 120pt row spacing, light grey RGB(0.92, 0.93, 0.94). Brand may override colour subject to a minimum-contrast guard.
- Content: template string `{brand_name} // {user} // {render_date}` (brand-customisable).
- Controls: `--watermark-user`, `--watermark-text`, `--no-watermark` (only valid if brand `security.watermark_min_level: L0`).

### 5.3 L2 тАФ XMP Metadata Watermark

Pikepdf writes XMP RDF with a custom `mdpdf:` namespace alongside Dublin Core:

| XMP Key | Source |
|---|---|
| `dc:creator` | brand_name |
| `dc:title` | document H1 or brand default |
| `pdf:Producer` | `md-to-pdf 2.0` |
| `xmp:CreatorTool` | `md-to-pdf 2.0` |
| `xmp:CreateDate` | ISO 8601 (frozen in deterministic mode) |
| `mdpdf:RenderId` | UUIDv4 (or derived hash in deterministic mode) |
| `mdpdf:RenderUser` | request.watermark.user |
| `mdpdf:RenderHost` | `sha256(hostname)[:16]` (no plaintext leak) |
| `mdpdf:BrandId` | brand id |
| `mdpdf:BrandVersion` | brand pack version |
| `mdpdf:InputHash` | `sha256(markdown_bytes)` |
| `mdpdf:WatermarkLevel` | `"L1+L2"` (always literal in v2.0) |

(HMAC tamper-protection field added in v2.3 alongside `policy.yaml > audit > hmac_secret`.)

### 5.4 Audit Log

**Format:** JSONL, two events per render (`render.start`, `render.complete` or `render.error`).

**Default location:** `~/.md-to-pdf/audit.jsonl`.

**Rotation:** daily, retain 90 days by default.

**Privacy:** input source is **never** copied; only `sha256` recorded. Output PDF likewise referenced by hash + path + size.

**Schema:** see appendix A.

**Permissions:**
- POSIX (Linux/macOS): `0640`, owner only.
- Windows: ACL restricted to current user.

(SIEM forwarding, log groups, system-mode `/var/log/...` placement land in v2.3 alongside `policy.yaml`.)

### 5.5 Mermaid Sandbox

**Input lint** (before dispatch):

- AST validator rejects: `click тАж callback`, raw HTML `<script>`, `style=`, `img src=http(s)://`.
- Length тЙд 50K chars.
- Node count тЙд 500.
- Subgraph nesting depth тЙд 10.

**Process isolation by renderer:**

- **Kroki** тАФ full HTTP isolation; Kroki container provides its own sandbox.
- **Puppeteer** тАФ `--disable-features=NetworkService,Extensions`; `file://` denied; 30s timeout; subprocess `prlimit` RAM cap 512MB on Linux.
- **pure-Python** тАФ no browser, no subprocess; safest by construction.

**Output limits:**

- Max output PNG тЙд 10MB (defends against svg-bomb attacks).
- Max concurrent Mermaid renders тЙд 4.

### 5.6 Temp File & Cleanup Policy

- All temp files under `tmpfs` (`/dev/shm` on Linux, `/tmp` on macOS, `%TEMP%` on Windows) when available.
- `cache/TempContext` context manager guarantees cleanup on success **and** exception.
- `atexit` hook scans for orphaned `mdpdf-tmp-*` directories on process exit.
- PDF output uses atomic write: `output.pdf.tmp.<random>` тЖТ `fsync` тЖТ `rename`.

| Code | Meaning |
|---|---|
| 0 | Success |
| 1 | Render failure (see stderr) |
| 2 | Configuration / argument error |
| 3 | Brand validation error |

### 6.2 Python API


## 7 ┬╖ Testing, CI, Versioning, Release

### 7.1 Test Pyramid

- **Unit** тАФ every module independently. AST transformers, brand schema, safe_paths, watermark layers each get their own test file. Coverage threshold тЙе 80% on `src/mdpdf/`.
- **Integration** тАФ full `Pipeline.render()` with real renderers (Kroki via dockerized fixture, Puppeteer in opt-in CI lane).
- **Golden** тАФ see ┬з7.2.
- **Stress** тАФ 500-page docs (RAM + time bounds), 50 Mermaid blocks (concurrency limits), 4K images (downsampling).

### 7.2 Golden Testing (Four Layers + Combinatorics)

Direct PDF byte diff is unstable due to embedded timestamps and ZLIB compression. Instead:

| Layer | What it catches | Update cost |
|---|---|---|
| AST diff | Parse changes (most stable) | Low |
| XMP diff | Post-process metadata | Low |
| Text-layer diff (`pdftotext` extract) | Content regressions | Low |
| Layout fingerprint (page count + per-page text-box count) | Layout regressions | Low |
| sha256 (deterministic mode only) | CI archival guarantee | Highest |

`tests/golden/` directory structure:

```
tests/golden/
тФЬтФАтФА v1_parity/                            # v1.8.9 fixtures, v2-rendered output
тФВ   тФЬтФАтФА uat-en/         {input.md, expected.ast.json, expected.xmp.json,
тФВ   тФВ                    expected.text.txt, expected.fingerprint.json}
тФВ   тФЬтФАтФА uat-zh/
тФВ   тФЬтФАтФА uat-cjk/
тФВ   тФЬтФАтФА uat-table/
тФВ   тФЬтФАтФА fenced-mermaid-smoke/
тФВ   тФФтФАтФА mermaid-noto-presets/
тФЬтФАтФА comprehensive/                        # full-scenario UAT (see ┬з7.2.1)
тФВ   тФФтФАтФА branch_ops_ai_robot_product_brief/
тФВ       {input.md, expected.ast.json, expected.xmp.json,
тФВ        expected.text.txt, expected.fingerprint.json,
тФВ        assets/}                          # high-res images, SVG referenced by input.md
тФФтФАтФА deterministic/
    тФФтФАтФА *.sha256                          # for --deterministic mode runs
```

Every fixture currently in `fixtures/uat-*.md` becomes a v1-parity golden test. Acceptance: AST diff zero, text-layer diff zero (excluding render-id substitutions), layout fingerprint within ┬▒2 page boxes per page (allows for sub-pixel layout differences but catches regressions).

#### 7.2.1 Comprehensive UAT Fixture (`branch_ops_ai_robot_product_brief.md`)

The small focused fixtures (`uat-en`, `uat-zh`, `uat-cjk`, `uat-table`, `fenced-mermaid-smoke`, `mermaid-noto-presets`) catch isolated regressions in single subsystems. They miss interactions between subsystems on the same page (e.g., a wide table immediately followed by a Mermaid diagram immediately followed by a code fence with line numbers, all under a CJK heading).

The new `fixtures/branch_ops_ai_robot_product_brief.md` is a single multi-page product brief (target 15тАУ25 PDF pages) that exercises every renderer code path in realistic prose context. **Creating this fixture is a v2.0 deliverable** (estimate: 1тАУ2 days of authoring + manual review).

**Required scenarios тАФ the fixture must include at least one of each:**

| Category | Required scenarios |
|---|---|
| **Document structure** | YAML front-matter (with `template:`, `brand:`, doc metadata); `## чЫох╜Х` table-of-contents pipe-table that exercises the TOC-promotion AST transformer; H1тАУH4 nesting with at least one run-on `# Part## Chapter` (heading-merge transformer); horizontal rule; document-end appendices |
| **Headings & paragraphs** | At least one CJK-only heading, one English-only, one mixed; long paragraphs (тЙе 200 chars) testing line-break with `wordWrap="CJK"`; paragraphs with inline `bold`, *italic*, `inline code`, [links](https://example.com), and inline images |
| **Lists** | Flat bullet list; flat numbered list; **3-level nested mixed list** (bullet > numbered > bullet); list item containing a fenced code block; list item containing an image |
| **Tables** | Narrow 3-column table; **wide 6+-column table** that triggers content-weighted column-width algorithm; table with CJK content (column-width with `wordWrap=CJK`); table with cells containing inline code and bold; table with empty cells |
| **Code fences** | Python (multi-line, тЙе 30 lines, exercises Pygments + line numbers + page break inside fence); TypeScript (different lexer); Bash (one-liner, tests narrow fence); YAML (config-style); plain text fence (no Pygments lexer); fence inside a list item (indented); a fence longer than `MDPDF_FENCED_MAX_LINES` to verify truncation behaviour |
| **Mermaid** | Flowchart (graph TD); sequence diagram; gantt chart; class diagram; one Mermaid block with a `title:` YAML header (exercises caption from title); one Mermaid block placed under a heading without a YAML title (exercises caption from heading); one Mermaid block that scales to ~30% of body height (exercises `KeepTogether`); one Mermaid block that scales to ~70% of body height (exercises page break + caption alignment) |
| **Images** | High-resolution raster image (тЙе 2400px wide, exercises auto-downsample to 300dpi); SVG image (exercises cairosvg тЖТ PNG); image with explicit width attribute; image inside a table cell; image immediately followed by a caption-style italic paragraph |
| **Block quotes & callouts** | Single-line block quote; multi-paragraph block quote; nested block quote |
| **Watermark interaction** | Document is long enough (15+ pages) that the L1 visible watermark appears on cover, body, code-page, image-page, table-page, and last page тАФ visually verifying the watermark survives all flowable types |
| **Fonts & i18n** | Body must contain mixed CJK (Chinese Simplified + Traditional) and English; at least one paragraph with Japanese kana to verify Noto SC fallback behaviour for non-SC CJK; at least one heading and one fenced code block containing CJK characters |
| **Brand integration** | Renders cleanly against the existing `brand_kits/` example brand; renders cleanly against a second neutral brand from `examples/brands/`; visible watermark contains brand name + user + render date in expected positions |

**Required image assets** (committed under `tests/golden/comprehensive/branch_ops_ai_robot_product_brief/assets/`):

- `architecture-overview-4k.png` (тЙе 2400px wide, ~3MB) тАФ exercises high-res raster + auto-downsample.
- `system-diagram.svg` тАФ exercises SVG тЖТ PNG via cairosvg.
- `screenshot-with-text.png` (~1080p) тАФ moderate-resolution raster.
- `inline-icon.png` (small, ~64px) тАФ used inline within a table cell.

The fixture is content-neutral: it describes a fictional "Branch Operations AI Robot" product brief covering market positioning, system architecture, technical specifications, API design, deployment plan, and roadmap. No real PII, no real customer/vendor names, no internal IP тАФ safe to commit publicly under Apache-2.0.

**Golden assertion for this fixture is layered:**

| Layer | Pass criterion |
|---|---|
| AST diff | Zero deviation from `expected.ast.json` (parser regressions) |
| Text-layer | Zero deviation from `expected.text.txt` (excluding render-id substitutions) |
